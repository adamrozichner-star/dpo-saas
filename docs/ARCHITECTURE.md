# Deepo Architecture (v3)

**Status:** authoritative architecture reference. Supersedes the prior 7-layer / knowledge-graph model, which is obsolete.
**Date:** 2026-06-21 · **Basis:** Product Vision v2 + live schema audit (`docs/audit-architecture-v3.md`).
**Audience:** engineering (Adam + Claude Code). English. Hebrew is used only for product terms where it is the real name of the thing.

---

## 1. Thesis

Deepo is not a knowledge base. It is an **operational loop** that manages live compliance over time, for every client, through the same cycle. Regulatory knowledge is fixed and standard. The hard, expensive part is the loop: figure out what applies, collect the truth from several parties, close the gaps, and keep it alive year after year.

The product is one thing: a **per-client Obligation Ledger** that the loop drives.

Storage is **relational** (Postgres). The graph is rejected: Amir's discovery gave us a certain schema (organizations, systems, vendors, databases, obligations, controls, tasks, contacts, events, evidence). The regulation catalog is a SQL reference table. The evidence chain is a relation, not a graph: `obligation` is fulfilled-by `evidence`. pgvector stays, but only as a semantic navigation aid over regulation text, never as the primary store.

---

## 2. The one-paragraph mental model

Every client moves through the **Five-C loop**. **Classify** scopes who the org is and what minimally applies. **Collect** gathers the real state of each obligation from multiple parties (the bottleneck: sysadmin and vendors). **Close** drives every gap to compliant (generate a document, assign a task, chase the right party). **Control** keeps it compliant over time (reminders, annual tasks, vendor chasing) and loops back to Collect when something expires. **Certify** proves it in one click (a full audit-readiness pack). Control is the heart: it is where the recurring value and recurring revenue live.

The loop does not invent obligations per client. It instantiates them from a fixed catalog. That is the scaling model in section 4.

---

## 3. Layers: reuse vs build

The audit makes the split clean. The catalog layer mostly exists. The per-client ledger layer is greenfield.

| Layer | What it is | State today |
|---|---|---|
| **L1 Universal** | Regulation catalog, classification rules, template structure, control engine. Built once, identical for every business. | **Largely exists.** `regulatory_documents` + `regulatory_sections` (+ pgvector), and the `hub_*` library (`hub_gap_rules.rule_dsl`, `hub_control_playbooks` with `cadence`, `hub_questions`, `hub_asset_templates`, `hub_document_templates`). Mature and v3-shaped. |
| **L2 Vertical profile** | Thin config per business type (typical systems, data types, questions in their language, which obligations it usually triggers). About one day of expert work. | **Partial.** The pieces to author profiles exist (`hub_asset_templates`, `hub_questions`), but there is no explicit "vertical profile" grouping object yet. See section 4. |
| **L3 Specific client** | The living Obligation Ledger for one business: its systems, its vendors, its obligation states, its control schedule. Auto-generated by the loop. | **Absent. This is the build.** No durable obligation/control/task/evidence rows exist. State is recomputed from jsonb blobs on every dashboard load and thrown away. |

The single biggest fact in the audit: **there is no persistent obligation storage and no near-equivalent.** Everything else in v3 sequences behind creating it.

## 4. Scaling model (how a new vertical costs one config, not one product)

Adding a vertical (say, a makeup shop) does not mean cataloguing makeup shops. We catalog two separate things, and most of it is shared:

- **L1 is identical for everyone.** The law does not change between a makeup shop and an accounting firm. The regulation catalog, the Five-C loop, the document structure, the control engine, the audit pack: all shared, forever.
- **L2 is a thin profile config per business type.** A profile points at: which `hub_asset_templates` are typical, which `hub_questions` to ask, in what language, and which catalog obligations those usually trigger. Roughly 80% comes pre-filled from L1. A new vertical may add one or two new asset types (a POS register, a loyalty club) to the **shared** asset catalog, and every future vertical can then reuse them. The tenth vertical may need no new work because its asset types already exist.
- **L3 is generated, never authored.** The client's living ledger is produced by running the loop against their profile.

**Implication for the data model:** a vertical profile is a config object, not code. Proposed: a `vertical_profiles` row (or a tagging convention on the existing `hub_*` library, keyed by a `vertical` value) that bundles "for business type X, these asset templates + these questions + these expected obligations." The expert (Roy / Amir) edits this through a simple console screen, no programmer. The exact shape is a small open design item, called out in the implementation plan.

## 5. The relational data model

### 5.1 Anchor and reuse (already live)

- **`organizations`** — the hub. 51/59 tables chain to it. Everything below is org-scoped. No change to its core role.
- **`data_recipients`** — the vendor table. Already carries `has_dpa`, `dpa_signed_date`, `dpa_expiry_date`, `compliance_verified`, `last_audit_date`, `risk_level`, `status`. This is the most v3-ready non-core entity.
  - **DECISION (1):** `data_recipients` becomes the single canonical home for vendors. The jsonb duplicates (`organization_profiles.third_parties`, `processing_activities.external_recipients`) are migrated in and then treated as derived/legacy. One home, three copies reconciled.
- **`cameras`** — the only asset type materialized as its own table. It already has `next_review_due_at`: a control-cadence pattern we generalize below.
- **`regulatory_documents` / `regulatory_sections`** — the regulation catalog (L1), versioned, sectioned, embedded. Reference data, read-only at runtime.
- **`hub_gap_rules`** — the catalog of obligation/gap definitions. `rule_dsl` (jsonb) is the declarative rule. Authored today, **never evaluated**. v3 makes the engine evaluate it to mint obligations.
- **`hub_control_playbooks`** — the catalog of controls, with `cadence` and `checklist`. Instantiated into per-org control rows.
- **`hub_questions` / `hub_asset_templates` / `hub_document_templates`** — the rest of L1: branching questionnaires, asset types, document bodies.

### 5.2 New ledger tables (L3, the build)

The center is `obligations`. Everything else hangs off it.

**`obligations`** — the heart. One row = one client-specific instance of a catalog requirement, carrying state.
- `id`, `org_id` → organizations
- `source_rule_id` → hub_gap_rules (which catalog requirement this instantiates), plus `source_version` for provenance
- `title`, `description`, `severity` (denormalized from the rule at mint time for stable display)
- `status` enum: `unknown` → `checking` → `in_treatment` → `compliant` → `expired`
- `triggered_by` (structured: what classification answer, asset, or collected fact created this obligation) plus `trigger_ref` (FK-ish reference to the asset/answer)
- `fulfilled_by_control_id` → controls (nullable: the control that keeps it satisfied), and `recurs_at` (when it next needs re-checking; null = one-time)
- `opened_at`, `status_changed_at`, `closed_at`
- evidence is linked via the `evidence` table (5.2), not a column here

**`controls`** — per-org instances of `hub_control_playbooks`. The recurring engine reads these.
- `id`, `org_id`, `source_playbook_id` → hub_control_playbooks (+ version)
- `cadence` (copied from playbook, overridable per org), `owner_role`, `next_due_at`, `last_completed_at`, `status`
- a control completing is what flips its linked obligations back toward compliant and sets the next `recurs_at`

**`tasks`** — actionable work items. Distinct from `dpo_queue`.
- **DECISION (3):** keep `dpo_queue` (live, 26 cols, DPO-console SLA work) as is. Add `tasks` for the broader loop: items assigned to the business owner, sysadmin, or a vendor, often delivered over a no-login link.
- `id`, `org_id`, `obligation_id` → obligations (nullable), `control_id` → controls (nullable)
- `assignee_actor` enum (`dpo` | `owner` | `sysadmin` | `vendor`), `assignee_contact_id` → contacts
- `title`, `status`, `due_at`, `access_link_id` → access_links (nullable), `completed_at`

**`evidence`** — the missing relation. Proof that an obligation is satisfied.
- `id`, `org_id`, `obligation_id` → obligations
- `kind` (`document` | `answer` | `attestation` | `external_file`)
- `document_id` → documents (nullable), `answer_ref` (nullable), `captured_at`, `captured_via` (which actor/link)
- this is the evidence chain: `obligation` ← fulfilled-by ← `evidence`. A simple FK, not a graph.

**`assets`** — generalizes the `cameras` precedent.
- **DECISION (2):** one per-org `assets` table, typed by `hub_asset_templates`. Asset-type-specific fields live in a `details` jsonb validated against the template, except where an asset type earns its own table (as `cameras` already has). New verticals add asset types to the shared catalog, not new tables.
- `id`, `org_id`, `asset_template_id` → hub_asset_templates, `name`, `details` (jsonb), `next_review_due_at`, timestamps
- `cameras` stays as its specialized table for now; a later pass decides whether to fold it under `assets` or keep the specialization.

**`systems`** and **`databases`** — promote from jsonb. Today they live in `processing_activities.systems_used` and `organization_profiles.databases`. In v3 they are either (a) their own thin tables or (b) asset types under `assets`. Leaning (b) for consistency, but this is a small open item resolved in the schema migration PR.

**`contacts`** — unify the scattered contact fields (`organizations.contact_*`, `data_recipients.contact_*`, `dpos`, `users`) into one org-scoped contacts entity, so a task or link can target a person. Lightweight: `id`, `org_id`, `name`, `email`, `phone`, `role`, `linked_entity` (optional pointer to a vendor/system).

**`events`** — a unified per-org timeline. Several append-only logs exist (`audit_logs`, `ropa_audit_log`, `agent_runs`, `incident_actions`, `notifications`). v3 does not rip these out. It adds one `events` table that the ledger writes to for material state changes (obligation opened, control completed, evidence captured, link used), so the audit pack and the dashboard timeline have one source. Existing logs stay for their current jobs.

### 5.3 What the five documents become

The five standard documents (privacy policy, database-definitions doc, security procedure / reg 4, systems-and-databases mapping, annual control plan) are **renders of the ledger**, not authored files. Each document row points at the obligations and assets it reflects, carries a version history, and a lifecycle (auto-generated → reviewed by DPO → approved by owner → published → next review). The audit pack is a render of the ledger plus these documents plus the evidence and control log.

## 6. The Five-C engine

Today there are two hardcoded `if/else` engines (`compliance-engine.ts`, `regulatory-engine.ts`) and a dead rule catalog (`hub_gap_rules.rule_dsl`, authored but never evaluated). v3 collapses this into one catalog-driven engine.

**The evaluator.** A deterministic function reads a client's facts (classification answers, assets, collected vendor/sysadmin responses) and evaluates `hub_gap_rules.rule_dsl` against them. For each rule that fires, it mints or updates an `obligations` row (with provenance back to the rule + version). For each rule that implies ongoing monitoring, it instantiates the matching `hub_control_playbooks` into a `controls` row with a `next_due_at`. This replaces both hardcoded engines. The DSL is the single source of inference truth, editable by experts without a deploy.

**The runtime.** `agent-invoke` (the existing Inngest function) is the execution surface. It is built but nothing emits `deepo/agent.invoke`. v3 wires the triggers: onboarding completion, a new asset added, a collected answer arriving, a control coming due, a document approved. Each trigger runs the relevant slice of the loop. The 5 generic CRUD skills grow the domain skills (mint obligation, open task, request evidence over a link, render document) that are currently marked "later phases."

**Determinism boundary (PC commitments).** Inference (which obligations apply) is deterministic, from the DSL. The LLM synthesizes language (document prose, plain-Hebrew explanations, chasing messages) and assists classification, but does not decide obligations. Every obligation points at the rule and version that minted it: that is the evidence chain on the inference side.

## 7. The four actors and the access model

The real work lives in the flows between four actors.

| Actor | Surface | Access |
|---|---|---|
| **DPO** (Roy / Amir) | Full console: all clients, compliance score each, a queue of judgment items (`dpo_queue`), sign-off and approval, full history. | Authenticated. |
| **Business owner** (Dana) | Light app: answers questions in plain language, sees their compliance state, approves prepared documents in one tap, notified only when truly needed. | Authenticated, light. |
| **Sysadmin** (external) | Short targeted questionnaire (backups, restore tested, 2FA, logs). The bottleneck. Most of the value is the automatic reminder and chase. | **No-login tokenized link. Zero info exposure.** |
| **Vendors** (Chashavshevet, Finbot, ...) | Upload a DPA or confirm wording, then submit an annual report. Chased automatically year over year. | **No-login tokenized link. Zero info exposure.** |

### The access-link primitive (new, nothing reusable exists)

The audit is explicit: nothing reusable exists, and the current public surfaces (`/trust/[slug]`, `/api/rights?action=get_org`) lean the wrong way, they disclose org identity and compliance data to whoever holds the identifier. v3 needs the opposite.

**`access_links`** — a single-purpose, scoped, revocable token.
- `id`, `token` (high-entropy, hashed at rest), `org_id`, `purpose` (`sysadmin_questionnaire` | `vendor_dpa` | `vendor_annual` | `dsar_intake`), `target_contact_id` (nullable), `expires_at`, `used_at`, `revoked_at`, `task_id` (nullable)
- A link resolves only to the **specific task or form** it was minted for. It never returns org name, compliance score, other clients, or any client data not strictly required for that one form. This is the hard constraint (CC-2): the link holder learns nothing about the organization.
- DSAR intake folds into this primitive: the requester's PII passes through to the business and is never stored at Deepo.

**Confirmed live CC-2 violation to retire (from the audit + the security hotfix):** today `data_subject_requests` stores subject PII in the primary DB (requester name, national ID / ת"ז, email, phone). This contradicts CC-2 and the vision's "pass-through, zero PII at Deepo" promise. The security hotfix gated the unauthenticated read so the PII is no longer publicly enumerable, but the data is still stored. v3 must convert DSAR to true pass-through: forward the request to the business, store at most non-identifying metadata (status, timestamps, the obligation/task it opened), and never persist the identifying fields. Retiring those columns is part of the access-link work (see implementation plan PR6).

**Other disclosure debt:** the `/trust/[slug]` and `get_org` paths disclose org identity to anyone holding the slug/UUID, the opposite of zero-exposure. These get re-evaluated against the access-link rule in the same PR.

## 8. Hard constraints (carried from the commitments)

- **CC-2 (zero subject PII):** no subject PII in the primary DB. Pass-through, tokenized, or aggregate only. The no-login links expose zero client info. This is non-negotiable and is the thing the whole insurance and positioning story rests on. **One confirmed live violation exists today:** `data_subject_requests` stores requester name, national ID, email, and phone (now access-gated by the security hotfix, but still stored). Retiring this to true pass-through is tracked in implementation plan PR6.
- **CC-4 (multi-expert governance with provenance):** every obligation and control traces to a catalog rule and version; every document carries who approved it and when.
- **PC commitments still hold:** latency targets, evidence chains, human review on uncertainty, zero measurable PII, costs under 5% of revenue.

---

## 9. Engineering posture (consequence of the drift finding)

The audit found severe drift: 24 of 59 live tables have no migration in the repo, 3 migrated tables do not exist live, and RLS/grants/functions could not be verified. Two rules follow, and they gate the implementation plan:

1. **The live database, not the repo, is the source of truth.** Before any v3 migration, capture a baseline `pg_dump` of the live schema (structure, RLS, grants, functions) and commit it. All v3 work starts from that baseline.
2. **Migrations are enforced from here.** No more out-of-band SQL-editor changes. Every schema change is a reviewed migration file, applied manually but tracked, verified post-commit in a separate query (not inside a rolled-back transaction). This is how the drift happened, and it stops now.

---

## 10. What this changes about the existing code

- The two hardcoded engines (`compliance-engine.ts`, `regulatory-engine.ts`) are superseded by the DSL evaluator. They are not deleted on day one: they are feature-flagged off as the ledger comes online, so the dashboard keeps working during the transition.
- `agent-invoke` stops being dormant: it gets triggers and domain skills.
- `work-plan` (currently serving stub task data) reads from the real `tasks` table.
- The jsonb stores (`organization_profiles.*`, `processing_activities.*`, `work_plans.tasks`) become migration sources, then legacy. They are read during migration, then the ledger is canonical.
